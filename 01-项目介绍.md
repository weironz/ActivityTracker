## Kafka 网站活动追踪详解

---

### 🏗️ 架构图

```
用户行为
  │
  ├─ 页面浏览 ──→ Topic: page-view
  ├─ 搜索行为 ──→ Topic: search
  ├─ 点击购买 ──→ Topic: purchase
  └─ 登录登出 ──→ Topic: auth
                      │
              Kafka Broker（中心枢纽）
                      │
        ┌─────────────┼─────────────┐
        ↓             ↓             ↓
   实时处理        实时监控      离线存储
 (推荐系统)      (异常告警)   (Hadoop/数仓)
```

---

### 📖 逐句拆解

**"将用户活动追踪管道重建为实时发布-订阅数据流"**

> 传统做法是用户行为直接写数据库，所有系统都查同一个库，耦合严重且扛不住高并发。Kafka 把它改造成**生产者发布、消费者订阅**的解耦模式。

---

**"每种活动类型对应一个主题"**

| Topic | 含义 |
|-------|------|
| `page-view` | 用户浏览了哪个页面 |
| `search` | 用户搜索了什么关键词 |
| `purchase` | 用户下单购买 |
| `click` | 用户点击了哪个按钮 |

每类行为**独立存储**，消费者按需订阅，互不干扰。

---

**"多种用例"具体是什么：**

| 用途 | 举例 |
|------|------|
| **实时处理** | 用户搜索"手机" → 立刻推送手机广告 |
| **实时监控** | 某页面报错率突增 → 立刻触发告警 |
| **离线分析** | 把一天的行为数据导入数仓 → 生成次日报表 |

同一份数据，**三个消费者同时消费，互不影响**，这是 Kafka 最大的优势。

---

**"数据量巨大"有多大：**

以电商网站为例，一个用户打开首页可能产生：
```
页面加载     → 1条
商品曝光×20  → 20条
广告曝光×5   → 5条
滚动行为×10  → 10条
─────────────────────
一次页面浏览  → ~36条消息
```
**1万并发用户 = 每秒几十万条消息**，传统消息队列根本撑不住，Kafka 天生为此设计。

---

### 💡 核心价值总结

```
解耦   → 前端埋点只管发，后端系统只管消费
削峰   → 流量高峰时 Kafka 缓冲，下游按自己节奏消费
复用   → 同一份数据同时供实时和离线系统使用
```
